## 2026-07-12 · 第五輪：日期與手機時間同步（修正 UTC 時差錯置一天）

### 問題
使用者回報網站時間與手機不同步、造成輸入錯誤。根因有二：
1. 全站取「今天」都用 `new Date().toISOString().split('T')[0]`——這是 **UTC 日期**。台灣（UTC+8）每天 00:00～08:00 之間會被算成**前一天**：日期欄位預設值錯一天、「今日」攝取加總也掛到錯的日子（驗證當下實測：本地 07-12、UTC 算出 07-11）
2. 日期欄位只在頁面載入時設定一次；手機頁面放過夜不關，隔天打開日期仍停在昨天

### 修正（`app.js`，資源版本 v=6）
- 新增 `getLocalDateStr()`：以裝置本地時區產生 YYYY-MM-DD，取代全部 4 處 `toISOString` 用法（預設日期、今日加總、AI 匯入、烹飪匯入）
- 新增 `syncDatesWithDevice()` 午夜自動翻頁：切回頁面時 + 每 30 秒檢查本地日期是否變更；變更時把**仍停在舊預設值**的日期欄位推進到新日期並重算今日加總；使用者**手動改過**的日期（補登舊資料）不會被動
- 打字中（焦點在輸入框）跳過同步，避免欄位在手指下被改動

### 驗證（本地 static server）
- `getLocalDateStr()` = 裝置本地日期（UTC+8 實測與 UTC 差一天的時段通過）
- 四個日期欄位預設值 = 本地今天
- 模擬跨午夜：自動預設的日期欄位推進到新日期 ✓、使用者手動選的補登日期保持不變 ✓、重複呼叫為 no-op ✓

## 2026-07-12 · 第四輪：完整營養素追蹤與跨頁整合

### 需求
1. 每日看板除了熱量外，要追蹤蛋白質與其他必要營養素的攝取量（依網路權威指引列出必要項目）
2. 整合各分頁，數據要有整體性（同一數字在每頁一致）

### 營養素研究依據（減脂期指引）
- **蛋白質**：阻力訓練者減脂期 2.0~2.4 g/kg（男 2.0、女 1.8 g/kg 落在建議範圍）
- **脂肪**：熱量赤字期 0.5~0.8 g/kg → 採 0.8 g/kg
- **碳水化合物**：蛋白質與脂肪扣除後的剩餘熱量 ÷ 4
- **膳食纖維**：男 30 g / 女 25 g（另一算法為 14 g / 1000 kcal）
- **鈉**：每日上限 2,300 mg（Dietary Guidelines for Americans / FDA）

### 修正
| 檔案 | 內容 |
|------|------|
| `app.js` | 新增 `getNutritionTargets()`（單一目標來源：cal/p/c/f/fiber/sodium/bmr）與 `computeTodayTotals()`（單一當日加總來源，'both' 均分）；重寫 `updateTodayProgress()` 同時驅動 Tab 1 進度條、Tab 4 收支卡與營養素明細；`foodLogs` 新增 `c/f/fiber/sodium` 欄位（AI 匯入與烹飪匯入都寫入完整營養）；AI 分析表新增纖維/鈉可編輯欄與總計；離線 `nutritionDB` 補 12 項食材纖維值；歷史飲食表新增碳水/脂肪欄 |
| `index.html` | Tab 1 每人新增「脂肪、膳食纖維」進度條（碳水改為真實值+動態目標，原本是攝取熱量×40% 的假估算+寫死 150/130g）；Tab 4 收支卡新增「🥗 今日營養素攝取進度」明細（蛋白質/碳水/脂肪/纖維/鈉五列迷你進度條）；AI 分析表 +纖維/鈉欄；資源版本 v=5 |
| `style.css` | 新增 `.nutrient-breakdown` 系列樣式；修正 `.quick-log-btn` 被 `.ai-action-btn { width:100% }` 撐爆 flex 列、把日期/體重輸入框擠成細條的桌面版 bug（`width:auto; flex:0 0 auto`） |
| `api/analyze.js` | Gemini prompt 增加 fiber(g)/sodium(mg) 欄位要求（照片與文字兩種模式） |

### 整體性設計
所有「今日攝取 vs 目標」數字統一出自 `getNutritionTargets()` + `computeTodayTotals()` 兩個函數：Tab 1 進度條、Tab 4 收支卡與營養明細、側欄目標熱量顯示全部同源，任何一處修改個人資料即全站同步重算。

### 驗證（本地 static server，桌面+375px 手機）
- AI 匯入 → `foodLogs` 完整寫入 `{cal:468, p:71, c:10, f:17, fiber:5, sodium:246}`
- Tab 1 進度條與 Tab 4 營養明細數字完全一致（71/170g、10/145g、17/68g、5/30g、246/2300mg）
- 歷史飲食表新欄位正常渲染；舊紀錄缺欄位顯示 `-`

## 2026-07-11 · 第一輪：程式碼研究與安全/正確性修正（commit `8ddccb9`）

### 研究方式
通讀 `app.js`（1811 行）、`index.html`、`style.css`、`api/` 三支 serverless functions 與全部文件，按嚴重程度列出問題清單。

### 發現的問題與修正

**1. 隱私：全站共用一把 KV 鑰匙（最嚴重）**
- 問題：`/api/load`、`/api/save` 固定讀寫 KV key `fitness_data`，無驗證、CORS 開 `*`。README 提到的 `settings.shareKey` 實際未實作——任何人打開網站就是同一份資料，可看到並覆寫體重與日誌
- 修正：實作專屬同步金鑰。前端產生 32 字元隨機金鑰（存 `localStorage`，附加於網址 `?key=`），雲端改存 `fitness_data_<key>`；`save.js` 強制驗證金鑰格式（`/^[A-Za-z0-9_-]{8,64}$/`）否則回 400；舊共用紀錄轉唯讀、僅供首次開啟時單次遷移；三支 API 移除 CORS `*`
- 驗證：不帶金鑰 POST `/api/save` → 400；陌生金鑰讀取 → 空白資料

**2. 「目標」欄位被「現況」覆寫（bug）**
- 問題：登錄今日體脂會把側邊欄「目標體脂 15%」蓋成當日量測值；登錄女生體重會把「目標體重 60kg」蓋成目前體重（共三處，疑似複製貼上錯誤）
- 修正：移除 `app.js` 中三處錯誤 sync（快速登記表單 ×2、`loadSharedData` / `deleteWeightItem`）
- 驗證：登錄 66.2kg 後目標體重仍為 60；登錄體脂 24.1% 後目標體脂仍為 15

**3. XSS 風險**
- 問題：食材名稱等使用者輸入直接用模板字串插入 `innerHTML`；配合舊版資料全站共享，存在跨使用者注入腳本的可能
- 修正：新增 `escapeHTML()`，套用於歷史表、AI 分析表、庫存清單所有使用者輸入
- 驗證：注入 `<img onerror>` 僅顯示為文字，不執行

**4. 單位不相容時盲目扣減**
- 問題：庫存記「2 包」、食譜要「250 g」時，系統判定庫存足，烹飪直接 2 − 250 = 0
- 修正：`getConvertedStock()` 不相容回傳 `null`；食譜顯示黃色 `⚠️ 單位不符` 警告；`deductStock()` 不相容時**不扣減**；AI 匯入庫存僅在單位可換算時合併同名項目
- 驗證：單位不符 badge 正確顯示、庫存維持不變

**5. 清空的庫存會復活**
- 問題：每次載入若庫存為空就強制塞回 4 筆範例資料
- 修正：僅在**整個資料庫全空**（全新安裝）時植入範例
- 驗證：清空庫存後重新載入維持 0 筆，其他資料不受影響

**6. 同步蓋寫緩解**
- 問題：整包覆寫、後寫者贏，兩支手機同時操作會互蓋
- 修正：切回分頁（`visibilitychange`）與每 90 秒自動重拉雲端資料；打字中跳過刷新

**7. 其他**
- 移除手機除錯用 `alert` 錯誤彈窗（改 console）
- README 修正過時檔名（`getSharedData.js` → 實際的 `load.js` / `save.js` / `analyze.js`）、補 shareKey 說明

### 部署注意事項（已完成）
升級後**其中一人先開啟網站**（觸發舊資料遷移），再用「📋 複製雙人共享網址」把含金鑰連結傳給伴侶，對方**必須用該連結開啟**，否則兩人金鑰分岔、資料分成兩份。

## 維護守則（給後續開發者 / AI Agent）

1. **每次修改都要在本文件新增紀錄**：日期、問題、修法、驗證方式、commit hash
2. 改動 `index.html` 引用的 `style.css?v=N` / `app.js?v=N` 版本號要遞增，避免手機舊快取
3. 使用者輸入插入 `innerHTML` 前必經 `escapeHTML()`
4. 側邊「目標」欄位（目標體脂/目標體重）不得被當日量測值覆寫
5. 手機（375px）為主要使用情境：改版後需驗證零橫向溢出、導覽可用、輸入框不觸發 iOS 縮放
6. 修改 `fitnessDB` 後依序呼叫 `saveSharedData()` + 對應 render 函數
7. `git push` 到 `main` 即自動觸發 Vercel 部署，部署後以 `curl` 驗證正式站內容
