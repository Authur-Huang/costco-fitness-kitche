- AI 分析表可橫向滑動（可視 271px / 內容 621px）

## 2026-07-11 · 第一輪：程式碼研究與安全/正確性修正（commit `8ddccb9`）

### 研究方式
通讀 `app.js`（1811 行）、`index.html`、`style.css`、`api/` 三支 serverless functions 與全部文件，按嚴重程度列出問題清單。

### 發現的問題與修正

**1. 隱私：全站共用一把 KV 鑰匙（最嚴重）**
- 問題：`/api/load`、`/api/save` 固定讀寫 KV key `fitness_data`，無驗證、CORS 開 `*`。README 提到的 `settings.shareKey` 實際未實作——任何人打開網站就是同一份資料，可看到並覆寫體重與日誌
- 修正：實作專屬同步金鑰。前端產生 32 字元隨機金鑰（存 `localStorage`，附加於網址 `?key=`），雲端改存 `fitness_data_<key>`；`save.js` 強制驗證金鑰格式（`/^[A-Za-z0-9_-]{8,64}$/`）否則回 400；舊共用紀錄轉唯讀、僅供首次開啟時單次遷移；三支 API 移除 CORS `*`
- 驗證：不帶金鑰 POST `/api/save` → 400；陌生金鑰讀取 → 空白資料

**2. 「目標」欄位被「現況」覆寫（bug）**
- 問題：登錄今日體脂會把側邊欄「目標體脂 15%」蓋成當日量測值；登錄女生體重會把「目標體重 60kg」蓋成目前體重（共三處，疑似複製貼上錯誤）
- 修正：移除 `app.js` 中三處錯誤 sync（快速登記表單 ×2、`loadSharedData` / `deleteWeightItem`）
- 驗證：登錄 66.2kg 後目標體重仍為 60；登錄體脂 24.1% 後目標體脂仍為 15

**3. XSS 風險**
- 問題：食材名稱等使用者輸入直接用模板字串插入 `innerHTML`；配合舊版資料全站共享，存在跨使用者注入腳本的可能
- 修正：新增 `escapeHTML()`，套用於歷史表、AI 分析表、庫存清單所有使用者輸入
- 驗證：注入 `<img onerror>` 僅顯示為文字，不執行

**4. 單位不相容時盲目扣減**
- 問題：庫存記「2 包」、食譜要「250 g」時，系統判定庫存足，烹飪直接 2 − 250 = 0
- 修正：`getConvertedStock()` 不相容回傳 `null`；食譜顯示黃色 `⚠️ 單位不符` 警告；`deductStock()` 不相容時**不扣減**；AI 匯入庫存僅在單位可換算時合併同名項目
- 驗證：單位不符 badge 正確顯示、庫存維持不變

**5. 清空的庫存會復活**
- 問題：每次載入若庫存為空就強制塞回 4 筆範例資料
- 修正：僅在**整個資料庫全空**（全新安裝）時植入範例
- 驗證：清空庫存後重新載入維持 0 筆，其他資料不受影響

**6. 同步蓋寫緩解**
- 問題：整包覆寫、後寫者贏，兩支手機同時操作會互蓋
- 修正：切回分頁（`visibilitychange`）與每 90 秒自動重拉雲端資料；打字中跳過刷新

**7. 其他**
- 移除手機除錯用 `alert` 錯誤彈窗（改 console）
- README 修正過時檔名（`getSharedData.js` → 實際的 `load.js` / `save.js` / `analyze.js`）、補 shareKey 說明

### 部署注意事項（已完成）
升級後**其中一人先開啟網站**（觸發舊資料遷移），再用「📋 複製雙人共享網址」把含金鑰連結傳給伴侶，對方**必須用該連結開啟**，否則兩人金鑰分岔、資料分成兩份。

## 維護守則（給後續開發者 / AI Agent）

1. **每次修改都要在本文件新增紀錄**：日期、問題、修法、驗證方式、commit hash
2. 改動 `index.html` 引用的 `style.css?v=N` / `app.js?v=N` 版本號要遞增，避免手機舊快取
3. 使用者輸入插入 `innerHTML` 前必經 `escapeHTML()`
4. 側邊「目標」欄位（目標體脂/目標體重）不得被當日量測值覆寫
5. 手機（375px）為主要使用情境：改版後需驗證零橫向溢出、導覽可用、輸入框不觸發 iOS 縮放
6. 修改 `fitnessDB` 後依序呼叫 `saveSharedData()` + 對應 render 函數
7. `git push` 到 `main` 即自動觸發 Vercel 部署，部署後以 `curl` 驗證正式站內容
